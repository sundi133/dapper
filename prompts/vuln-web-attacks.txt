<jwt_algorithm_checks>
Scope here is JWT verification algorithm handling only (OWASP A06). For every token-consuming endpoint:
- Locate the verify call: grep for jwt.verify(, jsonwebtoken, jwtVerify(, jose, verifyJwt, decode(. Read the exact site. jwt.decode() used in place of a verify call performs NO signature check at all — treat decode-as-verification as unrestricted-algorithm acceptance (worst case).
- Confirm an explicit algorithms allow-list is pinned, e.g. jwt.verify(token, key, { algorithms: ['HS256'] }). Missing/empty algorithms option, or algorithms derived from the token header, is the defect.
- alg=none: forge a header {"alg":"none"} with an empty signature and replay it against {{WEB_URL}}; a 200/authenticated response confirms the verifier accepts unsigned tokens.
- RS256->HS256 key confusion: if the server verifies RS256 with a public key, re-sign a token as HS256 using that public key as the HMAC secret and replay against {{WEB_URL}}; acceptance confirms key confusion.
- vulnerable_code_location MUST be the exact jwt.verify(...) / jwtVerify(...) site (file:line). Use vulnerability_type "JWT".
- Ownership boundary: JWT signing-secret management (hardcoded/weak/fallback secrets), token lifetime & revocation (long-lived/no deny-list/valid-after-logout), and localStorage/sessionStorage token storage are owned by the crypto agent (crypto-vuln). Do NOT duplicate those here; report only algorithm-restriction flaws in this queue.
</jwt_algorithm_checks>
