<quality_rules>
MANDATORY quality standards for the final report:

1. NO INTERNAL PATHS: Remove all file system paths like `/app/repos/...`, `/Users/...`, `/home/...`, `/tmp/...`. Replace with generic references like "the application source code" or "the configuration file".

2. NO ASSEMBLY ARTIFACTS: Remove all HTML comment markers (`<!-- BEGIN:... -->`, `<!-- END:... -->`, `<!-- SECTION:... -->`, `<!-- ASSEMBLY_METADATA ... -->`, `<!-- COMPLIANCE_MAPPING ... -->`).

3. CODE BLOCK LANGUAGE TAGS: Every code block must have a language identifier. Use `http` for HTTP requests/responses, `python` for Python, `javascript` for JS, `json` for JSON, `bash` for shell commands, `xml` for XML/HTML. If uncertain, use `text`.

4. NO EMPTY SECTIONS: Every section must have content. If a vulnerability category has no findings, explicitly state "No [category] vulnerabilities were identified during this assessment." Never leave a section with just a heading and no body.

5. NO TRUNCATED CONTENT: If exploitation steps appear cut off mid-sentence or mid-code-block, note "[Evidence truncated in source data]" rather than leaving broken content.

6. CONSISTENT SEVERITY FORMAT: Use exactly one of: Critical, High, Medium, Low, Informational. Map any variations (e.g., "CRITICAL", "critical", "Crit") to these canonical forms.

7. SELF-CONTAINED FINDINGS: Each vulnerability finding must be understandable on its own. Include the vulnerable endpoint, the attack description, and the impact without requiring the reader to cross-reference other sections.

8. NO HALLUCINATION: Do not invent findings, severity ratings, or exploitation evidence that does not exist in the source deliverables. If evidence is weak or incomplete, say so honestly.

9. PROFESSIONAL TONE: Write for a client audience. No informal language, no first-person narrative ("I tested..."), no tool-specific jargon that wouldn't appear in a professional pentest report.

10. FINDINGS TABLE ACCURACY: The consolidated findings table in section 2.2 must exactly match the detailed findings in section 5. Every ID in the table must appear in the details and vice versa.
</quality_rules>
